sm100_bf16_gemm_rs.cuh 完整源码逐行解析

总览：这个 kernel 做什么

BF16 GEMM + Reduce-Scatter 融合 kernel，在 SM100 (Blackwell) 上运行。

数学操作：C = A × B^T，其中 A∈R^{M,K}，B∈R^{N,K}，C∈R^{M,N}

Reduce-Scatter：每个 rank 只需最终输出 C 的一部分行（按 M 维度切分），所以 GEMM 的每个 tile 计算完就直接 NVLink 发给目标 rank，RS 线程在另一侧收集累加。

┌──────────────┐ Rank 0 视角: │ GEMM 计算 │ A[0..M/K], B │ C = A × B^T │ └──────┬───────┘ │ Epilogue ┌──────▼───────┐ │ NVLink 写到 │ │ 目标 rank 的 │ │ partial 缓冲 │ └──────┬───────┘ │ ┌──────▼───────┐ │ RS Warps: │ │ 等所有rank的 │ │ ready=1， │ │ 累加 partial │ │ → 写入 y │ └──────────────┘

线程角色分配

CTA 内 kNumRSThreads + 128 + 128 个线程，分 4 组：

Warp 范围角色标签
warp 0 ~ kNumRSWarps-1RS 收集线程：轮询 ready flag，累加各 rank 的 partial，写入最终输出 yRS
warp kGemmWarpBase (=kNumRSWarps)TMA Load Warp：通过 TMA 从全局内存加载 A/B tile 到 smemLOAD
warp kGemmWarpBase+1MMA Issue Warp：执行 UMMA FMA 指令，累加到 TMEM（仅 leader CTA）MMA
warp kGemmWarpBase+2TMEM 分配：初始化后空闲INIT
warp (RS+128)/32 起，共 128 线程Epilogue Warps：TMEM→smem→NVLink scatter 到远端 partialEPILOGUE
CTA 线程布局 (以 kNumRSThreads=128 为例, 共 384 线程): ├── warp 0..3 ── RS Warps (128 threads) ── 轮询+累加 ├── warp 4 ── TMA Load Warp ── 加载 A/B 到 smem ├── warp 5 ── MMA Issue Warp ── 执行 FMA (仅 leader CTA) ├── warp 6 ── TMEM Alloc ── 初始化后空闲 └── warp 7..10 ── Epilogue Warps (128) ── TMEM→smem→NVLink scatter

第6段：块调度逻辑 get_next_block (L132-151) SCHED

132     uint32_t stage_idx = 0, phase = 0;
133     auto advance_pipeline = [&](uint32_t& k_block_idx) {
134         ++ k_block_idx;
135         stage_idx = stage_idx == kNumStages - 1 ? 0 : stage_idx + 1;
136         phase ^= stage_idx == 0;
137     };
138     auto get_next_block = [&](uint32_t& block_idx, uint32_t& m_block_idx,
139                                uint32_t& n_block_idx, uint32_t& iter_idx) {
140         if (block_idx >= num_m_blocks * num_n_blocks)
141             return false;
142         const uint32_t m_rank_wave = block_idx / (num_m_blocks_per_rank * num_n_blocks);
143         const uint32_t rem = block_idx - m_rank_wave * num_m_blocks_per_rank * num_n_blocks;
144         const uint32_t local_m_block_idx = rem / num_n_blocks;
145         n_block_idx = rem - local_m_block_idx * num_n_blocks;
146         const uint32_t dst_rank = (m_rank_wave + 1 < kNumRanks) ?
147             (rank_idx + m_rank_wave + 1) % kNumRanks : rank_idx;
148         m_block_idx = dst_rank * num_m_blocks_per_rank + local_m_block_idx;
149         block_idx += kNumSMs;
150         ++ iter_idx;
151         return true;

调度算法详解

block_idx 是全局递增的块号，stride=kNumSMs。每次从 block_idx 反推出：

  • m_rank_wave：第几轮"遍历所有rank"（0,1,2...）
  • local_m_block_idx + n_block_idx：块内坐标
  • dst_rank：这个块的计算结果应该发给哪个 rank
假设 kNumRanks=4, num_m_blocks_per_rank=2, num_n_blocks=3: block_idx=0: wave=0, local_m=0, n=0 → dst_rank=(0+0+1)%4=1 block_idx=1: wave=0, local_m=0, n=1 → dst_rank=1 block_idx=2: wave=0, local_m=0, n=2 → dst_rank=1 block_idx=3: wave=0, local_m=1, n=0 → dst_rank=1 block_idx=4: wave=0, local_m=1, n=1 → dst_rank=1 block_idx=5: wave=0, local_m=1, n=2 → dst_rank=1 block_idx=6: wave=1, local_m=0, n=0 → dst_rank=(0+1+1)%4=2 ... block_idx=12: wave=2, local_m=0, n=0 → dst_rank=(0+2+1)%4=3 ... block_idx=18: wave=3, local_m=0, n=0 → dst_rank=(0+3+1)%4=0 (自己!)
设计精髓：每个CTA计算的tile，其M坐标属于dst_rank，所以计算完的结果刚好是dst_rank需要的partial数据。Epilogue直接NVLink发过去，不需要额外的数据搬运。

③ do_tmem_full_arrive: 仅在最后一个 K-block 时触发，通知 Epilogue "整个 tile 的 MMA 计算完毕"

33                                                 const uint32_t& m_block_idx = 0,
34                                                 const uint32_t& n_block_idx = 0) const {
35         const uint64_t base_offset = kNumBarrierSignalBytes + get_num_partial_bytes();

设计要点总结

  1. 计算通信完全重叠：Epilogue 做 scatter 的同时 RS warps 做 gather+reduce，零额外延迟
  2. 无锁同步：跨 rank 通过 ready_flag 的 acquire/release 语义 + NVLink barrier，无需 NCCL
  3. 调度决定通信：get_next_block 让每个 CTA 计算目标 rank 需要的数据，结果直接 NVLink 发送
  4. TMEM 双缓冲：kNumEpilogueStages=2，MMA 计算下一 tile 时 Epilogue 同时读上一个 TMEM 结果
  5. RS 逐元素串行：L312-327 逐元素遍历 + 逐 rank 串行累加，不是向量化 reduce。这是简化实现，未来可优化
  6. 自旋等待：RS 用 while(ld_acq_sys(ready_ptr)==0) 忙等，延迟最低但浪费算力
