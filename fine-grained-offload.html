Megatron 细粒度 Activation Offload

本文对 Megatron-LM 中 fine_grained_activation_offloading 模块进行逐层深度剖析，涵盖数据结构设计、前向/反向时序、Prefetch Pipeline、Warmup 录制机制及常见理解误区纠正。

背景：为什么需要 Activation Offload？

在大规模模型训练中，Pipeline Parallelism（流水线并行） 将模型按层切分到不同 GPU。前向传播时，每一层都需要保存激活值（activation）供反向计算梯度使用。这些激活值会占用大量 GPU 显存。

核心问题
显存瓶颈

大 batch / 深模型时，activation 占用超过显存上限

解决思路
GPU→CPU Offload

前向时把 activation 搬到 CPU，反向时再搬回 GPU

关键挑战
隐藏传输延迟

H2D / D2H 传输不能阻塞计算，需要异步重叠

本模块特色
细粒度控制

以子模块（attn_norm、qkv 等）为粒度，可独立开关

核心概念层次结构

整个 offload 系统有三个层次，从粗到细：

层次类对应粒度数量（示例）说明
1. Chunk ChunkOffloadHandler 一个 microbatch N microbatch → N 个 Chunk 包含该 microbatch 经过所有 layer 产生的全部激活
2. Group OffloadTensorGroup 一个 layer 的一个子模块 每个 Chunk 有若干 Group 如 attn_norm、qkv_linear、mlp 各是一个 Group
3. Tensor PyTorch Tensor 单个激活张量 每个 Group 有若干 Tensor 通过 autograd hook 自动捕获 save_for_backward 的张量

ChunkOffloadHandler

一个 microbatch 对应一个 ChunkOffloadHandler。它负责：

  1. 将前向传播保存的激活张量从 GPU 异步拷贝到 CPU（offload，释放显存）
  2. 在反向传播需要时，将它们从 CPU 异步拷贝回 GPU（reload）
  3. 以 Group 为粒度管理，每个 Group 对应一个子模块的激活集合

创建时机：在 PipelineOffloadManager.init_model_chunk_offload_handler() 中创建，由 GPTModel.forward() 在每个 microbatch 开始时（warmup 阶段）调用。

chunk0 (microbatch 0): ├── group[0]: "attn_norm" (layer 0 的 LayerNorm 激活) ├── group[1]: "qkv_linear" (layer 0 的 QKV 线性层激活) ├── group[2]: "attn" (layer 0 的 Attention 激活) ├── group[3]: "mlp_norm" (layer 0 的 MLP 前 Norm 激活) ├── group[4]: "mlp" (layer 0 的 MLP 激活) ├── group[5]: "attn_norm" (layer 1 的 ...) └── ... chunk1 (microbatch 1): ← 同样的结构 chunk2 (microbatch 2): ← 同样的结构

PipelineOffloadManager

全局单例，负责协调所有 Chunk 的 offload/reload 调度，维护前向和反向的 Chunk 队列。

Warmup 录制机制：录制一次，重放多次

💡
这是整个系统最核心的优化策略：类比 CUDA Graph——第一个 iteration 「录制」所有 Chunk/Group 结构，后续 iteration 直接「重放」，避免重复分配对象。

为什么可以这样做？

Pipeline Parallelism 的训练调度是确定性的：每个 iteration 的 microbatch 数量、前向/反向顺序、每层的子模块结构完全相同。因此第一次录制的结构完全可复用。

Warmup 阶段（第一个 iteration）：录制

1
创建 Chunk 对象：每个 microbatch 开始前调用 init_model_chunk_offload_handler()，创建 ChunkOffloadHandler，追加到 _cached_chunks_forward
2
收集 Group 结构：on_group_start_forward() 在 warmup 时创建 OffloadTensorGroup，记录每层每个子模块的 name
3
分析 Offload 策略：post_warmup_callback() 在 warmup 结束后分析哪些 Group 不需要 offload（如末尾几个 Group 用 _offload_margin 跳过），并为 CPU tensor pool 分配合适大小的内存
4
构建反向队列：通过 flush() + reversed() 将 Chunk 按反向执行顺序（LIFO）存入 _cached_chunks_backward

非 Warmup 阶段（后续 iteration）：重放

1
reset() 归零：_cur_forward_chunk = None，指针 _cached_chunks_index_forward = 0，每个 Chunk 调用 chunk.reset() 重置内部计数器（但不销毁对象）
2
init_model_chunk_offload_handler() 直接返回：不再创建新对象
3
通过 pop_forward_chunk(name) 按需取出缓存 Chunk：从 _cached_chunks_forward[index] 中按序复用
优化原因说明
避免重复分配OffloadTensorGroup、CUDA Event 等无需每次重新创建
CPU tensor pool 复用warmup 后 pool 中已有正确大小的 CPU tensor，后续零分配开销
Offload 策略只需分析一次post_warmup_callback 决策 _offload_margin 一次性完成
确定性保证复用同一组 Chunk 对象，前向/反向匹配关系不会出错
减少 GC 压力减少 Python 对象的创建与销毁

前向传播流程

Autograd Hook 注册：__enter__ / __exit__

这是「细粒度（fine-grained）」的核心机制。每个子模块计算被包裹在一个 with off_interface(..., name) as hidden_states: 语句中：

# transformer_layer.py 中的示例
with off_interface(self.offload_attn_norm, hidden_states, "attn_norm") as hidden_states:
    input_layernorm_output = self.input_layernorm(hidden_states)

with off_interface(self.offload_qkv_linear, hidden_states, "qkv_linear") as hidden_states:
    qkv_output = self.get_query_key_value_tensors(hidden_states, ...)

__enter__ 注册 PyTorch autograd 的 saved tensor hook：

def __enter__(self):
    cpu_offload.CPUOffloadEnabled = True
    torch._C._autograd._push_saved_tensors_default_hooks(
        self.on_save_for_backward,   # 前向：tensor → tensor_tag
        self.on_get_saved_tensor     # 反向：tensor_tag → tensor
    )

def __exit__(self, *args):
    cpu_offload.CPUOffloadEnabled = False
    torch._C._autograd._pop_saved_tensors_default_hooks()
Hook调用时机作用
on_save_for_backward(tensor)任何算子调用 ctx.save_for_backward(tensor) 时将 tensor 推入当前 Group（tensor_push），返回 tensor_tag
on_get_saved_tensor(tag)反向传播时任何算子访问 ctx.saved_tensors 时根据 tag 从 Group 取回 tensor（tensor_pop，可能需要 reload）
📌
每个 with 块只覆盖一个具体的子计算（一次 linear 或一次 norm），这就是「细粒度」的含义：不同子模块的 tensor 分到不同 Group，可独立决定是否 offload。

Chunk 定位：pop_forward_chunk(name)

每次 with 块开始（group_start）时，需要确定当前 layer 属于哪个 Chunk：

def pop_forward_chunk(self, name=None):
    if not self.do_offload:
        return self._cur_forward_chunk

    while not self._is_warmup and (
        self._cur_forward_chunk is None
        or self._cur_forward_chunk.finish_all_groups(name)
    ):
        if self._cached_chunks_index_forward >= len(self._cached_chunks_forward):
            self._cur_forward_chunk = None
            break
        self._cur_forward_chunk = \
            self._cached_chunks_forward[self._cached_chunks_index_forward]
        self._cached_chunks_index_forward += 1

    return self._cur_forward_chunk

finish_all_groups(name) 的语义：从当前 _offloaded_group_index 位置往后，找不到名为 name 的 Group 了，即当前 Chunk（microbatch）的同名子模块全部处理完毕，需要切换到下一个 Chunk（下一个 microbatch）。

while 循环条件含义动作
_cur_forward_chunk is Noneiteration 开始、还没取第一个 Chunk从列表取第一个
finish_all_groups(name) == True当前 Chunk 的同名 Group 已全部处理完切换到下一个 Chunk
finish_all_groups(name) == False当前 Chunk 还有同名 Group 待处理退出循环，返回当前 Chunk

Offload 流程：bulk_offload_group

在 on_group_commit_forward() 时，触发整个 Group 的批量异步 offload：

1
让 D2H stream 等待当前计算流完成（d2h_stream.wait_event(compute_event)）
2
在 D2H stream 上，遍历 Group 内所有 tensor，调用 offload(tensor)：将 GPU tensor 拷贝到 pinned CPU memory，将 _tensors[tag] 替换为 (device, cpu_backup, use_cpu_pool) 元组
3
记录 offload_event（用于后续 reload 时确认数据已到 CPU）
4
如果设置了显存立即释放（forced_released_tensors），立刻释放 GPU 端引用
⚠️
如果某个 tensor 小于 min_offloaded_tensor_size，tensor_need_offloading_checker 返回 False，该 tensor 不会被 offload，始终保留在 GPU 上。

反向传播流程

🔑
最重要的理解要点：在 autograd 反向图中，前向时 GroupStart → GroupCommit 的顺序在反向时完全反转为 GroupCommit.backward → GroupStart.backward。这导致两个方法的「pre/post backward」角色与名字直觉完全相反。
节点Forward 时角色Backward 时角色实际调用
GroupCommit.backward 层计算结束（post forward） Pre Backward 层反向开始之前 调用 on_group_commit_backward()，等待 reload 完成
GroupStart.backward 层计算开始（pre forward） Post Backward 层反向结束之后 调用 on_group_start_backward()，发起下一层的 prefetch reload

GroupStart.backward（Post Backward）：为下一层 Prefetch

def backward(ctx, grad_output):
    cpu_offload_handler = ctx.cpu_offload_handler
    cpu_offload_handler.on_group_start_backward()  # ← 触发 bulk_reload
    return grad_output, None, None, None

此时当前层的反向计算已经完成，它在 H2D stream 上异步发起下一层（前一层）的 tensor reload，让传输与后续计算重叠（overlap）。

GroupCommit.backward（Pre Backward）：等待 Reload 完成

在当前层反向计算开始之前，等待该层 tensor 的 reload event，确保 tensor 已在 GPU 上可用。

同时检查 cur_backward_chunk 是否是自己，不是则调用 pop_backward_chunk(name) 切换。

完整反向时序（以 Layer N 为例）

反向计算图的执行顺序（以两层为例）： GroupCommit(LayerN).backward ← [Pre BWD] 等待 LayerN 的 reload event 完成 ↓ LayerN 的反向计算 ← 使用已 reload 好的 tensor（tensor_pop 被触发） ↓ GroupStart(LayerN).backward ← [Post BWD] 在 H2D stream 上异步发起 LayerN-1 的 reload ↓ GroupCommit(LayerN-1).backward ← [Pre BWD] 等待 LayerN-1 的 reload event 完成 ↓ LayerN-1 的反向计算 ← 使用已 reload 好的 tensor ↓ GroupStart(LayerN-1).backward ← [Post BWD] 异步发起 LayerN-2 的 reload ↓ ...

tensor_pop 与 Tensor 状态机

反向计算中每次访问 ctx.saved_tensors 时，被 hook 拦截，调用 tensor_pop：

def tensor_pop(self, tensor_tag):
    tensor = self.offload_groups[group_id - 1].pop_tensor(tensor_tag)
    if isinstance(tensor, tuple):   # tuple = 还没 reload 回来
        tensor = self.reload(tensor)  # 同步 reload（fallback 路径）
    return tensor

一个 tensor 经历的状态变化：

① tensor_push： _tensors[tag] = tensor [原始 GPU Tensor] ↓ ② bulk_offload： tensor 大 → _tensors[tag] = (device, cpu, pool) [tuple 状态，已到 CPU] tensor 小 → _tensors[tag] = tensor [仍是 GPU Tensor，未 offload] ↓ ③ bulk_reload： tuple → reload → _tensors[tag] = tensor [正常 prefetch，变回 GPU Tensor] 未被调用 → _tensors[tag] 仍是 tuple [边界情况] ↓ ④ tensor_pop： isinstance(tensor, tuple)? → False：直接返回（正常路径） → True： 同步 reload 后返回（fallback 路径）

Prefetch Pipeline：计算与传输重叠

整个反向传播的设计核心是 Double Buffering / Pipeline Prefetch：当前层在计算梯度时，下一层的 tensor 正在被异步搬回 GPU，两者在不同 CUDA stream 上并行进行。

冷启动问题与解决方案

❄️
Prefetch 链条的起点没有「上一层」来触发它，因此第一个开始 backward 的 Chunk 的最后一层（最先 backward 的 Group）无法被提前 prefetch。
场景第一次 Reload 如何发生是否阻塞
第一个 Chunk 的最后一层 tensor_pop 中 tuple fallback：同步阻塞 reload ⚠️ 阻塞
后续 Chunk 的最后一层 前一个 Chunk backward 结束时调用 pre_reload_last_layer 提前 prefetch ✅ 无阻塞
同一 Chunk 内的非最后一层 上一层 GroupStart.backward（Post BWD）触发 bulk_reload prefetch ✅ 无阻塞

pre_reload_last_layer 机制

在 bulk_reload() 中，当前 Chunk 自己的 Group 都 reload 完了之后，会「顺手」帮下一个 Chunk 的最后一层提前 reload：

def bulk_reload(self):
    if len(self._groups_to_reload) > 0:
        self.bulk_reload_group()          # 正常 prefetch 本 Chunk 下一层
    else:
        # 自己的活干完了，帮下一个 Chunk 的最后一层热身
        next_chunk = PipelineOffloadManager.get_instance().front_backward_chunk()
        if (next_chunk is not None
            and next_chunk._offloaded_group_index == next_chunk._max_group_size):
            next_chunk.pre_reload_last_layer()
Chunk0 backward 时序示意： Chunk0-LayerN: [无 prefetch，tensor_pop 同步 reload 阻塞] GroupStart(LayerN).bwd → bulk_reload → 为 LayerN-1 prefetch ──────────────┐ │ H2D stream 传输中... Chunk0-LayerN-1: │ GroupCommit(LayerN-1).bwd → wait_reload_event → ✅ tensor 已就绪 ◄─────────┘ [LayerN-1 反向计算，无阻塞] GroupStart(LayerN-1).bwd → bulk_reload → 为 LayerN-2 prefetch ... Chunk0-Layer1: GroupStart(Layer1).bwd → _groups_to_reload 为空 → pre_reload_last_layer(Chunk1) ────────────────────┐ │ H2D stream 传输中... Chunk1-LayerN: │ GroupCommit(LayerN).bwd → wait_reload_event → ✅ tensor 已就绪 ◄───────────┘ [无阻塞！]

反向 Chunk 定位

反向传播中，每次 GroupCommit.backward（pre backward）开始时，需要确认当前正在处理的是哪个 Chunk：

def pop_backward_chunk(self, name=None):
    self._cur_backward_chunk = None
    for idx, handler in enumerate(
        self._cached_chunks_backward[self._cached_chunks_index_backward:]
    ):
        self._cached_chunks_index_backward += 1
        if not handler.is_empty_chunk(name):
            self._cur_backward_chunk = handler  # 找到第一个包含该 name 的 Chunk
            break
    assert self._cur_backward_chunk is not None

front_backward_chunk(name) 是只读版本（peek），不推进指针，用于 prefetch 时查看下一个 Chunk。

已知理解误区纠正

以下是阅读源码时容易产生的理解错误，已在原文中标出并纠正：

❌ 误区 1：时序图中 on_group_start_backward 放在 backward 最前面

错误描述：认为 on_group_start_backward 是「pre backward」，在当前层 backward 开始前触发 bulk_reload。

正确理解：on_group_start_backward 是 GroupStartFunction.backward，而 Forward 时 GroupStart 在 GroupCommit 之前，所以 Backward 时顺序反转，GroupStart.backward 在 GroupCommit.backward 之后执行。它是 Post Backward，在当前层反向结束后为下一层 prefetch。

❌ 误区 2：认为 on_group_commit_backward 是「post backward」

错误描述：因为 commit = 结束，以为它在 backward 结束时调用。

正确理解：on_group_commit_backward 是 GroupCommitFunction.backward，由于 backward 顺序反转，它是 Pre Backward（在当前层反向计算之前），负责等待 reload 完成，确保当前层的 tensor 可用。

❌ 误区 3：认为 on_group_start_backward 的 bulk_reload 是为当前层服务的

错误描述：以为 GroupStart.backward 发起 reload 是为当前 group 补货。

正确理解：GroupStart.backward 是 Post Backward，此时当前层的反向已完成，它触发的 bulk_reload 是为下一个（前一个 layer）的 Group 提前 prefetch，实现 pipeline 重叠。

❌ 误区 4：认为 finish_all_groups 返回 True 时循环停止

错误描述：误以为 True = 找到了目标，循环结束。

正确理解：True = 「这个 Chunk 里所有同名 Group 都处理完了」= 活干完了，需要继续循环找下一个 Chunk。False = 「还有同名 Group 没处理」= 停下来用这个 Chunk。

🎯
核心设计哲学：
  • 「录制一次，重放多次」：Warmup 阶段构建结构，后续复用，类比 CUDA Graph
  • 「细粒度钩子」：以子模块为粒度注册 autograd hook，精确控制哪些 tensor 被拦截
  • 「流水线 Prefetch」：利用独立 CUDA stream，让 H2D 传输与 GPU 反向计算并行
  • 「前向 Post BWD 发起，后向 Pre BWD 等待」：GroupStart.backward（post）prefetch，GroupCommit.backward（pre）wait
