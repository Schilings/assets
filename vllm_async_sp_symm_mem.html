vLLM Async SP — Symmetric Memory 深度解析

第三层运行时：PyTorch Symmetric Memory 如何实现通信与计算的流水线重叠

1. 什么是 Symmetric Memory（对称内存）？

想象你有一个 8 GPU 的节点。通常每个 GPU 只能访问自己的显存——如果想读别人的数据，得先"发消息"让对方拷贝过来。

Symmetric Memory打破了这堵墙：所有 GPU 在各自的显存中分配一块相同虚拟地址的缓冲区，然后通过 NVLink 的 multicast（多播）能力，让任何一个 GPU 都能直接读写其他 GPU 上的对应位置。

一句话概括：所有 rank 的 buffer 拥有相同的虚拟地址，且互相知道彼此的物理位置，使得 GPU 可以直接发起远程读/写/multicast 操作，无需 CPU 介入。

2. 两套对称内存路径详解

这就是你问的核心问题。vLLM 中确实有两条完全不同的对称内存路径，它们使用不同的底层机制，服务于不同的场景：

路径 A PyTorch Symmetric Memory

torch.distributed._symmetric_memory

  • 由 PyTorch 原生提供
  • 核心能力：融合算子的流水线重叠
  • 用于 fused_all_gather_matmul（聚合+GEMM，输入侧）
  • 用于 fused_matmul_reduce_scatter（GEMM+散播，输出侧）
  • 用于 fused_all_gather_scaled_matmul / fused_scaled_matmul_reduce_scatter（FP8 变体）
  • 用于 multimem_all_reduce_ / two_shot_all_reduce_
  • 通信+计算在同一个算子内部按 chunk 流水线执行

路径 B NCCL Symmetric Memory

2.1 路径 A：PyTorch Symmetric Memory

初始化流程

# 文件: vllm/distributed/device_communicators/symm_mem.py
import torch.distributed._symmetric_memory as torch_symm_mem

class SymmMemCommunicator:
    def __init__(self, group, device, ...):
        # ① 分配对称内存缓冲区 — 所有 rank 相同虚拟地址
        self.buffer = torch_symm_mem.empty(
            self.max_size // self.dtype.itemsize,
            device=self.device, dtype=torch.bfloat16
        )

        # ② 所有 rank 同步，交换地址信息，建立跨 rank 互访
        handle = torch_symm_mem.rendezvous(
            self.buffer, self.group.group_name
        )

        # ③ 确认硬件支持 NVLink multicast
        assert handle.multicast_ptr != 0
关键步骤解读：
  1. torch_symm_mem.empty() — 在每个 GPU 上分配一块 buffer，PyTorch 保证所有 rank 的 buffer 虚拟地址相同
  2. rendezvous() — 这是一个集合通信（collective），所有 rank 互相交换 buffer 的物理地址，然后注册到 NVLink 的 multicast 映射表
  3. multicast_ptr != 0 — multicast 是 NVLink 的一种硬件能力，允许一个写操作同时更新所有 GPU 的对应地址。如果返回 0 说明硬件不支持

All-Reduce 的两种算法

def all_reduce(self, inp, *, out=None):
    # 1. 将输入拷贝到对称内存 buffer
    self.buffer[:inp.numel()].copy_(inp.view(-1))

    if use_multimem:
        # 算法一：NVLink Multicast 一步 All-Reduce
        #   每个 rank 向自己的 buffer 写入本地数据，
        #   然后通过 multicast 一次操作完成 reduce+broadcast
        torch.ops.symm_mem.multimem_all_reduce_(
            self.buffer[:inp.numel()], "sum", self.group.group_name
        )
    else:
        # 算法二：两步 Reduce + Broadcast
        #   Step 1: reduce-scatter（每个 rank 负责一部分求和）
        #   Step 2: all-gather（每个 rank 广播自己算好的部分）
        torch.ops.symm_mem.two_shot_all_reduce_(
            self.buffer[:inp.numel()], "sum", self.group.group_name
        )

    # 2. 从对称内存 buffer 拷贝结果到输出
    out.copy_(self.buffer[:inp.numel()].view(out.shape))
何时用哪种算法？
  • multimem_all_reduce_ — world_size ∈ {4,6,8} (SM90) 或 {6,8} (SM100) 时使用，利用 NVLink multicast 硬件，一步到位
  • two_shot_all_reduce_ — world_size 较小（如 2）时使用，两步完成

2.2 路径 B：NCCL Symmetric Memory

这条路径的思想完全不同——它不创建新的通信原语，而是给 NCCL 换一个内存分配器。

原理

NCCL 通信时，数据通常需要先从 PyTorch 的 cudaMalloc 分配的内存拷贝到 NCCL 内部缓冲区，才能发起通信。ncclMemAlloc 分配的内存天然可被 NCCL 直接访问，省去这一步拷贝。

3. 融合算子的流水线重叠 — 最精彩的部分

这是路径 A 的核心魔力。让我们深入看看 fused_all_gather_matmul 和 fused_matmul_reduce_scatter 是如何在 一个算子内部 实现通信与计算的流水线重叠的。

3.1 两种融合模式 — 一对对称的"阴阳"

Async SP 有两个核心融合算子，它们在 Transformer 层中交替出现，形成完美的对称：

模式 A: AllGather + GEMM

通信先行，计算跟后

AG chunk 0
→
GEMM chunk 0

用途：QKV/MLP 的输入侧

算子：fused_all_gather_matmul

先聚合各 rank 分片 → 再对完整数据做 GEMM

模式 B: GEMM + ReduceScatter

计算先行，通信跟后

GEMM chunk 0
→
RS chunk 0

用途：Attention/MLP 的输出侧

算子：fused_matmul_reduce_scatter

先做 GEMM → 再将结果分片散播到各 rank

3.2 为什么要融合？

不融合的情况（模式 A 为例）：

AllGather（等所有数据到齐）
→
GEMM（等通信完成再算）

通信和计算是串行的，总时间 = 通信时间 + 计算时间。浪费！

融合后的情况（模式 A）：

AG chunk 0
→
GEMM chunk 0
&
AG chunk 1
→
GEMM chunk 1
&
AG chunk 2
→
GEMM chunk 2
→
...

通信和计算流水线重叠，总时间 ≈ max(通信, 计算) + 少量气泡。

融合后的情况（模式 B）：

GEMM chunk 0
→
RS chunk 0
&
GEMM chunk 1
→
RS chunk 1
&
GEMM chunk 2
→
RS chunk 2
→
...

GEMM 算完一个 chunk 就立刻散播，同时继续算下一个 chunk。计算与通信同样流水线重叠。

3.5 流水线分块的工作原理 — 模式 A（真实源码）

下面是 PyTorch torch.distributed._symmetric_memory 中的真实源码，不是伪代码！

3.5.1 核心流水线引擎 _pipelined_multi_all_gather_and_consume

这是 fused_all_gather_matmul 的底层流水线实现。它的工作是：先让每个 rank 把自己的 shard 拷贝到对称内存，然后轮流从远程 rank 的 buffer 拉数据，每拉到一个 shard 就立刻执行 shard_consumer（即 GEMM）。

# 文件: torch/distributed/_symmetric_memory/__init__.py
# 核心函数: _pipelined_multi_all_gather_and_consume

def _pipelined_multi_all_gather_and_consume(
    shard,              # list[Tensor]: 本 rank 的输入 shard(s)
    shard_consumer,     # Callable: 每收到一个远程 shard 就调用它（做 GEMM）
    ag_out,             # list[Tensor]: all-gather 的完整输出（预分配）
    group_name,         # 进程组名
    ag_out_needed=True,
):
    # ① 计算需要的 P2P buffer 大小，获取对称内存 workspace
    p2p_workspace_size_req = 0
    for x in shard:
        p2p_workspace_size_req += x.numel() * x.element_size()
    symm_mem = get_symm_mem_workspace(group_name, min_size=p2p_workspace_size_req)
    group_size = symm_mem.world_size
    rank = symm_mem.rank

    # ② 全局 barrier — 确保所有 rank 都就绪
    symm_mem.barrier(channel=0)
    backend_stream = _get_backend_stream()
    backend_stream.wait_stream(torch.cuda.current_stream())

    # ③ 获取指向本 rank 对称内存 buffer 的视图
    def get_p2p_bufs(remote_rank):
        offset_bytes = 0
        bufs = []
        for x in shard:
            buf = symm_mem.get_buffer(     # ← 核心！跨 rank 读取 buffer
                remote_rank,                #   指定远程 rank
                x.shape,                    #   形状
                x.dtype,                    #   数据类型
                storage_offset=offset_bytes // x.element_size(),
            )
            bufs.append(buf)
            offset_bytes += buf.numel() * buf.element_size()
        return bufs

    local_p2p_bufs = get_p2p_bufs(rank)  # 本 rank 的 buffer

    # ④ 先把本地 shard 拷贝到本 rank 的对称内存 slot
    copy_shard(dst=local_p2p_bufs, src=shard)
    symm_mem.barrier(channel=1)  # 等所有 rank 都完成拷贝
    backend_stream.wait_stream(torch.cuda.current_stream())

    # ⑤ 先消费本地 shard（不需要跨 rank 传输）
    shard_consumer(shard, rank)

    # ⑥ ===== 流水线核心：轮流拉远程 shard 并消费 =====
    for step in range(1, group_size):
        # 双流交替：奇偶 step 用不同 CUDA stream，实现计算/通信重叠
        if step % 2 == 0:
            stream = torch.cuda.current_stream()
        else:
            stream = backend_stream

        remote_rank = (step + rank) % group_size
        remote_p2p_bufs = get_p2p_bufs(remote_rank) # 远程 rank 的 buffer

        with stream:
            # ⑥a 从远程 rank 的对称内存 P2P 拷贝到本地
            #    底层走 NVLink，不经过 CPU！
            copy_shard(dst=shards[remote_rank], src=remote_p2p_bufs)

            # ⑥b 立刻对该 shard 执行 GEMM
            shard_consumer(shards[remote_rank], remote_rank)

    # ⑦ 同步 & 最终 barrier
    torch.cuda.current_stream().wait_stream(backend_stream)
    symm_mem.barrier(channel=0)
关键 API 解读：
  • symm_mem.get_buffer(remote_rank, shape, dtype) — 返回一个 Tensor 视图，指向 remote_rank 的对称内存。底层通过 NVLink 直接访问远程 GPU 的显存，零拷贝！
  • copy_shard(dst, src) — 底层是 dst.copy_(src)，当 src 是远程 buffer 时，CUDA 驱动自动走 NVLink P2P DMA 传输
  • symm_mem.barrier(channel) — 基于 signal pad 的轻量级 barrier。不同 channel 独立，可以嵌套使用
  • 双流交替（step%2 切换 stream）— 让两个 CUDA stream 交替执行 "P2P copy + GEMM"，实现计算与通信的流水线重叠

上面的流水线函数不直接做 GEMM，而是把 GEMM 逻辑抽象为 shard_consumer 回调。下面是 fused_all_gather_matmul 中的消费者实现：

3.5.3 Native 异步 TP：_async_input_mm + Signal 机制

当 M 维度较小（2048 < M ≤ 4096）时，PyTorch 还有一个更激进的路径：用 signal + async GEMM kernel 让 GEMM 自己等数据，完全不需要 host 端同步！

# 文件: torch/distributed/_symmetric_memory/__init__.py
# _fused_all_gather_matmul_native — 最激进的路径

def _fused_all_gather_matmul_native(A_shard, B, group_name):
    symm_mem = rendezvous(A_shard, group_name)
    rank = symm_mem.rank
    world_size = symm_mem.world_size

    # 预分配输出和 signal 数组
    A = A_shard.new_empty(A_shard.shape[0] * world_size, A_shard.shape[1])
    A_signals = torch.zeros(world_size, dtype=torch.uint32, device=A_shard.device)
    A_shards = A.chunk(world_size)

    # ① 把本地 shard 拷贝到输出 buffer，并写 signal 通知 GEMM kernel
    A_shards[rank].copy_(A_shard)
    _SymmetricMemory.stream_write_value32(A_signals, rank, 1)
    #   ↑ cuStreamWriteValue32: GPU 端原子写，在 write 之前
    #     会发出 system-level fence，确保 copy_ 的数据对 GEMM 可见

    # ② 启动异步 GEMM — 它会自己等 signal！
    out = torch.ops.symm_mem._async_input_mm(A, B, A_signals, rank)
    #   ↑ 这是一个自定义 CUDA kernel，内部逻辑：
    #     while signals[chunk_id] == 0: spin_wait;
    #     // data is ready, do GEMM on this chunk
    #     chunk_id++;

    # ③ 在后台流中，继续从远程 rank 拷贝数据并写 signal
    for step in range(1, world_size):
        src_rank = (rank + step) % world_size
        src_buf = symm_mem.get_buffer(src_rank, A_shard.shape, A_shard.dtype)
        with backend_stream:
            A_shards[src_rank].copy_(src_buf)  # P2P 拷贝
            _SymmetricMemory.stream_write_value32(A_signals, src_rank, 1)
            # ↑ 写 signal → 告诉 _async_input_mm "这个 chunk 准备好了"

    # ④ 最终同步
    current_stream.wait_stream(backend_stream)
    symm_mem.barrier()
    return A, out
Signal 机制 vs Barrier：
  • barrier(channel) — 基于 signal pad 的跨 rank 同步，所有 rank 必须都到达才能继续
  • stream_write_value32 + _async_input_mm — GPU 端细粒度通知。GEMM kernel 内部 spin-wait 等 signal，不需要 host 端介入，延迟更低
  • Signal 机制特别适合小 batch 场景，因为 barrier 的开销相对更显著

3.5.4 Multicast 快速路径

当硬件支持 NVLink multicast 且 M ≤ 2048 时，还有一条更快的路径：

# 文件: torch/distributed/_symmetric_memory/__init__.py
# _multimem_all_gather_matmul — 利用 multicast 硬件

def _multimem_all_gather_matmul(A_shard, Bs, group_name):
    group = resolve_process_group(group_name)
    A_shape = (A_shard.shape[0] * group.size(), *A_shard.shape[1:])

    # 从对称内存 workspace 获取 buffer
    symm_mem = get_symm_mem_workspace(group_name, A_shape.numel() * A_shard.element_size())
    A = symm_mem.get_buffer(symm_mem.rank, A_shape, A_shard.dtype)

    # 一步完成 all-gather！底层使用 NVLink multicast 硬件
    torch.ops.symm_mem.multimem_all_gather_out(A_shard, group_name, A)

    # 对完整数据做 GEMM
    return [torch.matmul(A, B) for B in Bs]
Multicast vs 流水线：Multicast 路径不做分块流水线，而是利用硬件一次完成 all-gather（一次写，所有 rank 都看到），然后再做 GEMM。当数据量小（M ≤ 2048）时，multicast 的延迟比分块流水线更低，因为少了多次 kernel launch 的开销。

3.6 流水线分块的工作原理 — 模式 B（真实源码）

再看模式 B (GEMM + ReduceScatter)的真实源码。它使用 _pipelined_produce_and_all2all 流水线引擎，方向与模式 A 相反：先算 GEMM，再把结果通过 all-to-all 散播出去。

3.6.1 核心流水线引擎 _pipelined_produce_and_all2all

# 文件: torch/distributed/_symmetric_memory/__init__.py
# 核心函数: _pipelined_produce_and_all2all

def _pipelined_produce_and_all2all(
    chunk_producer,     # Callable[[dst_rank, out_tensor], None]: 生产一个 chunk
    output,             # Tensor: all-to-all 输出（预分配）
    group_name,         # 进程组名
    out_chunk_dim=0,
):
    out_chunks = output.chunk(group_size, dim=out_chunk_dim)
    p2p_workspace_size_req = out_chunks[0].numel() * out_chunks[0].element_size() * 2
    symm_mem = get_symm_mem_workspace(group_name, min_size=p2p_workspace_size_req)
    group_size = symm_mem.world_size
    rank = symm_mem.rank

    symm_mem.barrier(channel=0)
    backend_stream = _get_backend_stream()
    backend_stream.wait_stream(torch.cuda.current_stream())

    # 准备两个本地 P2P buffer，双缓冲！
    # step [i] 的结果写入 buf_0 时，远程 rank 正在从 buf_1 拉数据
    local_p2p_buf_0 = get_p2p_buf(rank, 0)
    local_p2p_buf_1 = get_p2p_buf(rank, 1)

    # ===== 流水线核心 =====
    for step in range(1, group_size):
        remote_rank = (rank - step) % group_size
        if step % 2 == 0:
            stream = torch.cuda.current_stream()
            p2p_buf, remote_p2p_buf = local_p2p_buf_1, get_p2p_buf(remote_rank, 1)
        else:
            stream = backend_stream
            p2p_buf, remote_p2p_buf = local_p2p_buf_0, get_p2p_buf(remote_rank, 0)

        with stream:
            # ① 执行 GEMM，结果直接写入本地 P2P buffer
            chunk_producer((rank + step) % group_size, p2p_buf)

            # ② barrier: 确保所有 rank 都完成了这一步的计算
            symm_mem.barrier(channel=step % 2)

            # ③ 从远程 rank 的 P2P buffer 拉数据到本地输出
            out_chunks[remote_rank].copy_(remote_p2p_buf)

            # ④ 再次 barrier: 确保所有 rank 都读完了
            #    （本 rank 的 P2P buffer 才能被下一步覆盖）
            symm_mem.barrier(channel=step % 2)

    # ⑤ 处理本 rank 自己的 chunk
    chunk_producer(rank, out_chunks[rank])

    torch.cuda.current_stream().wait_stream(backend_stream)
    symm_mem.barrier(channel=0)
模式 B 的关键设计：
  • 双缓冲（buf_0 / buf_1）— 写 buf_0 的同时远程 rank 可以从 buf_1 读，避免读写冲突
  • 两次 barrier — 第一次等所有 rank 算完，第二次等所有 rank 读完。两次 barrier 之间就是 P2P 拷贝窗口
  • 计算→通信→计算→通信的流水线 — chunk_producer（GEMM）和 copy_（P2P 传输）交替执行

3.6.2 GEMM 生产者：chunk_producer 回调

# 文件: torch/distributed/_symmetric_memory/__init__.py
# _fused_matmul_reduce_scatter_impl 中的 chunk_producer 定义

# scatter_dim != last_dim 的路径
def chunk_producer(rank, out):
    # A_shards[rank]: 输入 A 按 world_size 分的第 rank 个分片
    # B: 权重矩阵
    # out: 写入目标（本地 P2P buffer）
    torch.ops.aten.mm.out(A_shards[rank], B, **kwargs, out=out)

# scatter_dim == last_dim 的路径（特殊优化）
def _chunk_producer(rank, out):
    # B 按输出维度分片，每个 rank 对应不同的 B_shard
    torch.ops.aten.mm.out(A_flat, B_shards[rank], **kwargs, out=out)

# 调用流水线引擎
_pipelined_produce_and_all2all(
    chunk_producer,
    stacked_partials,   # all-to-all 输出（未 reduce 的部分和）
    group_name,
)
# 最后对 stacked_partials 做 reduce (sum/avg)
result = reduce_fn(stacked_partials.view(*leading_dims, -1), dim=scatter_dim)
模式 A vs B 的微妙区别：
  • 模式 A (AG+GEMM)：通信是"拉"模型 — 本地主动从远程读数据，到了就算
  • 模式 B (GEMM+RS)：通信是"推+拉"模型 — 本地算完写入 P2P buffer，远程来拉；本质是 all-to-all
  • 模式 B 需要额外的 reduce 步骤 — all-to-all 之后还要 sum/avg 各 rank 的部分和
  • 两者在 Transformer 层中交替出现，形成"收→算→推→算→收→..."的循环

3.7.1 Signal Pad — 对称内存的同步基础设施

每个对称内存分配都附带一块 signal pad——一块 P2P 可访问的特殊内存，用于跨 rank 同步。

# Signal pad 的大小可以通过 API 调整
# 默认大小足以支持常规的 barrier 操作
torch.distributed._symmetric_memory.set_signal_pad_size(1024 * 1024)  # 1MB

# barrier(channel) 的底层逻辑：
#   1. 每个 rank 写 signal pad 的自己的 slot (stream_write_value32)
#   2. 每个 rank spin-wait 等所有其他 rank 的 slot 都被写入
#   3. 所有 rank 都看到所有 signal 后，barrier 返回
#   4. channel 参数决定使用 signal pad 的哪个偏移区域

3.7.2 stream_write_value32 的特殊性

这个函数是 _fused_all_gather_matmul_native 的核心魔法：

# cuStreamWriteValue32 的语义：
#   1. 在执行写操作之前，发出一个 system-level memory fence
#   2. 这保证了 fence 之前的所有 memory 操作（如 copy_）对其他 GPU 可见
#   3. 然后原子地写一个 uint32 值到指定地址
#
# _async_input_mm kernel 端的逻辑：
#   while (signals[chunk_id] == 0) { /* spin wait */ }
#   // signal != 0 意味着对应 chunk 的数据已经准备好
#   // system-level fence 保证了数据可见性
#   do_gemm_on_chunk(chunk_id);

# 在 CUDA graph capture 模式下，stream_write_value32 不可用
# 替代方案：memset32
if not torch.cuda.is_current_stream_capturing():
    _SymmetricMemory.stream_write_value32(A_signals, rank, 1)
else:
    _SymmetricMemory.memset32(A_signals, offset=rank, val=1, count=1)

3.7.3 Low-Contention 通信 — 不抢占 SM 的方式

当通信需要与密集计算重叠时，低争用（low-contention）版本牺牲一点通信吞吐，换取更好的重叠效率：

# 文件: torch/distributed/_symmetric_memory/__init__.py
# _low_contention_all_gather — 使用 Copy Engine 而不是 SM

def _low_contention_all_gather(tensor, group_name):
    symm_mem = rendezvous(tensor, group_name)
    rank, world_size = symm_mem.rank, symm_mem.world_size
    output = tensor.new_empty(tensor.shape[0] * world_size, *tensor.shape[1:])
    chunks = output.chunk(world_size)

    with _get_backend_stream():
        if not input_is_symm_mem:
            local_buf = symm_mem.get_buffer(rank, tensor.shape, tensor.dtype)
            local_buf.copy_(tensor)  # ← SM-based copy

        # Pull: 从每个远程 rank 拉数据
        symm_mem.barrier()
        for step in range(world_size):
            remote_rank = (rank - step) % world_size
            src_buf = symm_mem.get_buffer(remote_rank, tensor.shape, tensor.dtype)
            chunks[remote_rank].copy_(src_buf)  # ← Copy Engine P2P
        symm_mem.barrier()

# _low_contention_reduce_scatter — Push + offline reduction
def _low_contention_reduce_scatter(tensor, reduce_op, group_name):
    # Push: 把数据推到远程 rank 的对称内存
    for step in range(world_size):
        remote_rank = (rank - step) % world_size
        dst_buf = workspace.get_buffer(remote_rank, ...)
        dst_buf.copy_(chunks[remote_rank])  # ← Copy Engine P2P
    # Offline reduction: 在本地对所有 rank 推过来的数据做 sum
    ret = a2a_res.unflatten(0, (world_size, -1)).sum(dim=0)
Low-Contention 的设计哲学：常规通信（如 NCCL）会将整个集合通信融合成一个 kernel，占用大量 SM。当通信与计算重叠时，这会抢夺计算所需的 SM 资源。Low-contention 版本拆分成多个小 kernel，优先使用 Copy Engine（DMA 引擎，不占 SM），只在必要的 reduction 步骤使用 SM，从而减少与计算的争用。

4. all_reduce 的决策路径 — 谁先谁后？

当一个 all_reduce 请求到来时，CudaCommunicator.all_reduce() 会按照以下优先级选择通信后端：

①
NCCL SymmMem all_reduce (路径B) — all_reduce_symmetric_with_copy
大张量 & 小张量时优先
②
Custom AllReduce — vLLM 自研 P2P kernel
中等张量时优先
③
PyTorch SymmMem all_reduce (路径A) — multimem_all_reduce_
fallback
④
NCCL 原生 all_reduce
最终 fallback
注意：路径 A 和路径 B 的 all_reduce 不会同时用在同一个 all_reduce 调用。它们是互斥的后备选项。但路径 A 的融合算子（fused_all_gather_matmul 等）是独立于这条 all_reduce 路径的——它们在图优化阶段就已经被替换了，不会走到 all_reduce 的决策路径。

6. NCCL SymmMem 的智能调度

路径 B 虽然没有流水线重叠的能力，但它通过零拷贝在特定场景下能获得显著加速。vLLM 基于实测数据做了一个自适应选择策略： 

# 文件: vllm/distributed/device_communicators/all_reduce_utils.py

# 基于 H100 和 GB200 实测的配置
# 8 GPU 时：
#   2K - 16K:  PyNCCL-symm 胜出 (1.35x - 1.48x 加速)
#   32K - 64K: Custom_AR 胜出
#   128K - 1G: PyNCCL-symm 胜出 (1.12x - 6.14x 加速)

def should_nccl_symm_mem_allreduce(world_size, input_tensor):
    tensor_size = input_tensor.nbytes

    # 小张量 → NCCL symm_mem (低延迟)
    # 中等张量 → Custom AR (P2P 更快)
    # 大张量 → NCCL symm_mem (高带宽)
    if world_size == 4:
        return tensor_size <= 16K or tensor_size >= 512K
    elif world_size == 8:
        return tensor_size <= 16K or tensor_size >= 128K

7. 总结：两套路径的协作关系

两套对称内存路径不是替代关系，而是互补关系：

场景 使用的路径 效果
Async TP 融合 GEMM+通信 路径 A 通信+计算流水线重叠，延迟 ≈ max(comm, comp)
独立 all_reduce (大/小张量) 路径 B 零拷贝，省去 copy-in/copy-out 开销
独立 all_reduce (中等张量) Custom AR P2P 直传，无需额外缓冲区
图捕获 (CUDA Graph) 两者都支持 路径 B 需要 pool ID 切换；路径 A 的融合算子天然支持

终极效果：在一个 Transformer 层中，QKV/GMLP 的 GEMM 计算与 AllGather/ReduceScatter 通信通过路径 A 的融合算子完美重叠，残余的独立 all_reduce 通过路径 B 零拷贝加速。两层合力，将通信延迟几乎完全隐藏在计算背后。
