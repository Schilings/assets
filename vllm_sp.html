从 SP 集成方案到 AsyncTP 协同，再到底层图改写机制
参考来源：微信公众号「LLM高性能计算」 - elrond-g
整理时间：2026-04-30

第一部分：vLLM 中的 SP 集成方案

1.1 为什么需要 SP？

SP 的集成是为了解决 张量并行（TP）的冗余开销。

问题描述
TP 模式下，每次 AllReduce 后所有卡都会持有完整的 [num_tokens, hidden_size] 张量， 导致 RMSNorm、残差相加等逐元素操作在所有卡上重复执行，浪费计算资源和显存。 一个 32 层模型一次前向推理需要 64 次 AllReduce，冗余开销显著。

SP 的核心设计目标

目标说明
通信量持平AllReduce = ReduceScatter + AllGather，拆分后总通信量无增加
计算/显存降为 1/TT 为 TP 卡数，单卡只处理 1/T 的 token
正交性与 TP 在 hidden 维度的切分正交，可叠加
对模型透明不需要修改任何模型代码

1.2 核心架构：编译期图改写方案

vLLM 没有要求模型适配 SP 版本，而是选择在 torch.compile 的计算图阶段 通过 Pattern Matcher 自动完成子图替换，核心组件是 SequenceParallelismPass。
  • 优化逻辑集中在编译层，可动态决策是否开启 SP
  • 与 TP、AsyncTP 完全正交，可协同生效

1.3 配置与自动降级逻辑

核心配置项（vllm/config/compilation.py）

配置项说明
enable_spSP 总开关，要求 TP>1；hidden_size 小于阈值会自动关闭
sp_min_token_numtoken 数下限，低于阈值退化回普通 TP，避免 Scatter/Gather 固定开销超过节省的计算量

自动降级判定（get_sequence_parallelism_threshold）

  • 仅支持 CUDA 设备，目前只对 H100（计算能力 90） 配置了阈值
  • H100 上要求模型 hidden_size ≥ 8192
  • LLaMA-7B（4096）、Qwen2-7B（3584）等会被自动跳过
min_token_num = (min_per_gpu_size_mb × tp_size × MiB) / (hidden_size × element_size)

H100 上 min_per_gpu_size_mb = 8MB，hidden_size 越大、TP 卡数越多，门槛越低。

1.4 核心实现：SequenceParallelismPass

4 种待匹配的子图 Pattern

Pattern 类型适用场景
FirstAllReduceRMSNormPatternBlock 入口的无残差场景：AllReduce → RMSNorm
MiddleAllReduceRMSNormPatternBlock 中间带残差融合：AllReduce → 残差相加 + 融合 RMSNorm
FirstAllReduceRMSNormStaticFP8Pattern无残差场景的 Static FP8 量化变体
MiddleAllReduceRMSNormStaticFP8Pattern带残差场景的 Static FP8 量化变体

自定义通信算子注册

在 vllm/distributed/parallel_state.py 中通过 direct_register_custom_op 把 ReduceScatter、AllGather 注册为 torch.ops.vllm 自定义算子，并实现 fake impl 做编译期 shape 推理：

  • ReduceScatter：fake impl 输出张量 token 维（dim=0）为输入的 1/T
  • AllGather：fake impl 输出张量 token 维为输入的 T 倍

子图替换逻辑

无残差场景
Input → AllReduce → RMSNorm → Output
替换为：
Input → ReduceScatter → 本地 RMSNorm → AllGather → Output
单卡仅计算 1/T 的 token 对应的 Norm。
带残差场景
ReduceScatter 后得到当前 rank 的 token 段，全局残差也切分为对应段， 残差以 scatter 状态跨层传递，直到需要完整序列时才执行 AllGather， 单卡显存占用降低 T 倍。

1.5 运行时适配机制

1

is_residual_scattered_for_sp 函数

位于 vllm/v1/worker/utils.py，判断当前残差张量是完整状态还是 scatter 状态。 piecewise 编译模式下，仅当输入 token 数在预编译的 size 列表中时，才保持残差为 scatter 状态。

2

上游调度适配

Scheduler 会将输入 token 数 pad 为 TP 卡数的整数倍， 保证 ReduceScatter 在 token 维（dim=0）可整除。

开启示例

LLM(
    model="...",
    tensor_parallel_size=4,
    compilation_config={
        "pass_config": {
            "enable_sp": True,
            "fuse_gemm_comms": True,   # 同时开启 AsyncTP，最大收益
        },
    },
)
收益参考
SP 本身收益约 5~15%，配合 AsyncTP 后收益显著提升（15-30%）。

维度SPCP

第二部分：AsyncTP 协同 —— 通信与计算如何 overlap

2.1 问题本质：表面看不能 overlap

经典的 SP + TP 流程：

Rank 0: [A0] ──┐ Rank 1: [A1] ──┼── AllGather ──> [A0|A1|A2|A3] ──> GEMM(A, W) ──> [Y0|Y1|Y2|Y3] ──> ReduceScatter ──> [Y_local] Rank 2: [A2] ──┤ Rank 3: [A3] ──┘
朴素实现的串行依赖
1. GEMM 必须等 AllGather 完全完成（拿到所有数据）
2. ReduceScatter 必须等 GEMM 完全完成
→ 通信和计算纯串行，无法 overlap

2.2 关键洞察：矩阵乘法的"行可分性"

这是 AsyncTP 的核心数学基础：

Y = A · W

其中 A 是 [N, K]，W 是 [K, M]，Y 是 [N, M]。

关键性质
Y 在行维度（N 维）上完全独立：
Yi = Ai · W
第 i 行只依赖 A 的第 i 行 → 拿到 A 的某些行就能立即算出 Y 的对应行。
视角对比
视角依赖粒度能否 overlap
张量级AllGather 整体 → GEMM 整体❌ 串行
行级A[i] → Y[i]，各行独立✅ 可流水线

2.3 all_gather_matmul 的实现原理

分块流水线

假设 TP=4，每个 rank 持有 A 的一个分片 Ai（shape [N/4, K]）。

朴素实现（串行）

通信:
AllGather (T_comm)
计算:
GEMM (T_gemm)
总耗时 = T_comm + T_gemm

AsyncTP 实现（流水线）

通信:
recv A1
recv A2
recv A3
计算:
GEMM A0
GEMM A1
GEMM A2
GEMM A3
↑ 第一块立即开始，无需等通信；最后只剩一块 GEMM
总耗时 ≈ max(T_comm, T_gemm) + T_gemm/T

核心 Triton kernel 伪代码

def all_gather_matmul(a_local, weight):
    # 1. 在 symmetric memory 上分配输出 buffer
    a_full = symm_mem.empty([N, K])
    a_full[my_rank * chunk : (my_rank+1) * chunk] = a_local

    # 2. 触发 P2P 写入信号（无阻塞）
    symm_mem.barrier_async()

    # 3. 启动 GEMM kernel，按 chunk 流水线
    output = torch.empty([N, M])
    for step in range(world_size):
        src_rank = (my_rank + step) % world_size

        # 等待该 chunk 到达（轻量级 spin-wait）
        symm_mem.wait_signal(src_rank)

        # 立刻 GEMM 这一块
        chunk_a = a_full[src_rank * chunk : (src_rank+1) * chunk]
        output[src_rank * chunk : (src_rank+1) * chunk] = chunk_a @ weight

    return output
关键点：
  • 第一个 chunk（自己的）不需要任何通信，立即可算
  • 后续每个 chunk 的通信和前一个 chunk 的计算同时进行
  • 信号同步是细粒度的，不是 NCCL 那种粗粒度全局 barrier

2.4 fused_matmul_reduce_scatter 的实现原理

ReduceScatter 的可分性

Yrank=iout = Σj=0..T-1 Yjin[i·N/T : (i+1)·N/T]

每个输出分片只依赖所有 rank 上对应位置的那一块，与其他位置无关。

流水线实现

计算:
GEMM 块0
GEMM 块1
GEMM 块2
GEMM 块3
通信:
send 块0
send 块1
send 块2
send+reduce
总耗时 ≈ T_gemm + T_comm/T

每算完一块就立即异步发送，与下一块的 GEMM 重叠。

2.5 为什么需要 Symmetric Memory？

传统 NCCL AllGather 的局限
NCCL AllGather 是一个"黑盒大操作"：你调用 dist.all_gather()， 它内部完成所有通信，返回完整张量。你没有办法在通信进行中读取"已经到达的部分"。

torch.distributed._symmetric_memory

PyTorch 2.x 引入的关键基础设施 —— 对称共享内存：

# 每个 rank 在同一段虚拟地址上分配 buffer
sym_buffer = symm_mem.empty(shape, device='cuda')

# 所有 rank 可以通过 P2P 直接读写其他 rank 的 buffer
# 无需经过 NCCL 的同步语义
remote_data = sym_buffer.get_remote(rank=1)  # 直接拿到 rank 1 的数据
这样 GEMM kernel 可以主动去"拉"数据， 而不是被动等待通信完成 → 实现细粒度的流水线。

2.6 收益分析

Speedup = (Tcomm + Tgemm) / (max(Tcomm, Tgemm) + min(Tcomm, Tgemm)/T)
场景加速比
T_comm ≈ T_gemm（通信计算均衡）接近 2×
T_comm ≪ T_gemm（计算密集）几乎完全隐藏通信
T_comm ≫ T_gemm（通信受限）受带宽限制，加速有限
实际场景： H100 + NVLink 环境下，大 batch + 大 hidden_size 的 prefill 阶段， 通信和计算量级相当，AsyncTP 通常能带来 15-30% 的端到端加速。

为什么 SP 是 AsyncTP 的前提？

❌ 没有 SP

图结构：GEMM → AllReduce

AllReduce 是"先 reduce 再广播"，无法做行级流水线

✅ 有 SP

图结构：AllGather → GEMM → ReduceScatter

这两个通信原语天然支持行维度的分块流水线

所以 SP 不仅是"消除冗余计算"，更重要的是 把通信原语改造成了适合 overlap 的形式 —— 这才是 vLLM 把它们设计成正交且必须先开 SP 的根本原因。

第三部分：图改写原理 —— 这个"图"到底是什么？

3.1 这个"图"是什么？—— PyTorch 的计算图

从代码到图

def transformer_block(x, w1, residual):
    out = torch.matmul(x, w1)
    out = all_reduce(out)
    out = out + residual
    out = rms_norm(out)
    return out

在 torch.compile 内部，这段代码会被转换成一张"图"：

x ──────┐ ▼ w1 ──────> [matmul] │ ▼ [all_reduce] │ ▼ residual ──┐ [add] ◄─────────────┘ │ ▼ [rms_norm] │ ▼ output
计算图（Computation Graph）的两个核心元素：
  • 节点（Node）：一个算子（op），如 matmul、all_reduce、add、rms_norm
  • 边（Edge）：数据流（张量）从一个算子流向另一个算子

3.2 这张图从哪来？—— torch.compile 的工作流程

你的 Python 代码
↓ ① TorchDynamo 追踪
FX Graph (中间表示)
↓ ② 图优化 Pass （vLLM 在这里插手！）
优化后的 FX Graph
↓ ③ Inductor 后端编译
高效的 CUDA Kernel

TorchDynamo 的神奇之处

  • 不是静态分析代码
  • 而是实际运行一遍，用 CPython 的 frame evaluation API 拦截字节码
  • 记录所有 tensor 操作，构建 FX Graph

FX Graph 长什么样？

graph():
    %x : [num_users=1] = placeholder[target=x]
    %w1 : [num_users=1] = placeholder[target=w1]
    %residual : [num_users=1] = placeholder[target=residual]
    %matmul : [num_users=1] = call_function[target=torch.matmul](args=(%x, %w1))
    %all_reduce : [num_users=1] = call_function[target=torch.ops.vllm.all_reduce.default](args=(%matmul,))
    %add : [num_users=1] = call_function[target=operator.add](args=(%all_reduce, %residual))
    %rms_norm : [num_users=1] = call_function[target=torch.ops.vllm.rms_norm.default](args=(%add,))
    return (%rms_norm,)

每行就是图中的一个节点，%xxx 是中间结果（边）。 这就是 vLLM 要改写的那张图。

3.3 子图与 Pattern 是什么？

子图（Subgraph）

大图里的一小块相连节点。vLLM 关心的子图：

input │ ▼ [all_reduce] │ ▼ residual [add] ◄────── │ ▼ [rms_norm] │ ▼ output

这个 4 节点子图在一个 Transformer 模型里可能出现几十次。

Pattern（模式）

= 子图的"模板"。就像正则表达式 \d{3}-\d{4} 能匹配 "123-4567"， Pattern 是对子图结构的描述。

class MiddleAllReduceRMSNormPattern:
    def pattern(self, x, residual, weight):
        """描述要匹配的子图结构"""
        x = torch.ops.vllm.all_reduce(x)
        x = x + residual
        out = torch.ops.vllm.rms_norm(x, weight)
        return out, x

    def replacement(self, x, residual, weight):
        """描述要替换成的新子图"""
        x = torch.ops.vllm.reduce_scatter(x)       # 替换 all_reduce
        residual = chunk(residual, my_rank)        # 残差也切片
        x = x + residual
        out = torch.ops.vllm.rms_norm(x, weight)   # 本地 rms_norm
        return out, x
核心就两个函数：
• pattern()："我要找这种形状的子图"
• replacement()："找到后换成这种形状"

3.4 替换是怎么操作的？

算法：子图同构匹配

这是计算机科学经典的 子图同构（Subgraph Isomorphism） 问题： 给定大图 G 和小图 P，找到 G 中所有与 P 结构相同的部分。

from torch.fx import subgraph_rewriter

# 一行代码完成"找 + 换"
subgraph_rewriter.replace_pattern(
    gm=my_fx_graph,           # 大图（整个模型）
    pattern=pattern_fn,       # 要找的模式
    replacement=replacement_fn # 替换成什么
)

匹配过程示例

... [matmul_1] ──> [all_reduce_1] ──┬──> [add_1] ──> [rms_norm_1] ──> [matmul_2] ──> ... ▲ residual_1 匹配器扫描每个节点： 尝试 1: matmul_1 起点 → 不是 all_reduce，跳过 尝试 2: all_reduce_1 起点 ✓ 节点类型匹配 all_reduce ✓ 下游是 add ✓ add 的另一个输入是 residual ✓ add 下游是 rms_norm → 匹配成功！ ...

替换过程：图的"外科手术"

1
切断旧节点连接：把 all_reduce → add → rms_norm 摘下来
2
插入新节点：按 replacement() 创建 reduce_scatter、chunk、新 add、新 rms_norm
3
重新连接边：旧的输入接到新的 reduce_scatter，新的 rms_norm 接到旧的下游
4
删除旧节点：被孤立的旧节点被垃圾回收

手术前后对比

手术前

[matmul_1] │ ▼ [all_reduce_1] │ ▼ residual_1 [add_1] ◄────── │ ▼ [rms_norm_1] │ ▼ [matmul_2]

手术后

[matmul_1] │ ▼ [reduce_scatter] │ ▼ [add_new] ◄── [chunk] ◄── residual_1 │ ▼ [rms_norm_new] │ ▼ [matmul_2]

整个图被原地修改了，但对外接口（输入输出）保持不变， 所以模型代码完全不用改。

3.5 为什么用图改写而不是改模型代码？

❌ 思路 A：修改模型代码

class LlamaDecoderLayer:
  def forward(self, x, residual):
    # SP 版
    x = reduce_scatter(x)
    residual = chunk(residual, my_rank)
    x = x + residual
    x = rms_norm(x)
    return x, residual
问题：
  • vLLM 支持几十种模型，每种都要改
  • FP8 量化又是一套
  • 小模型/短序列还要写分支
  • 模型代码非常臃肿

✅ 思路 B：图改写（vLLM）

# 模型代码永远干净：
x = all_reduce(x)
x = x + residual
x = rms_norm(x)

# 优化在编译期自动完成
# - 需要 SP？注册 Pass
# - 需要 AsyncTP？再注册一个
# - 需要 FP8？Pattern 自动匹配
优点：
  • ✅ 模型代码零侵入
  • ✅ 优化可组合
  • ✅ 动态决策是否启用
  • ✅ 易于单独测试
这是编译器领域的经典思路 —— 把优化和语义分离。 LLVM、GCC 做 CPU 优化的方式就是这样：你写的 C 代码很简单，编译器通过几十个 Pass 把它变得高效。

3.6 vLLM 中 Pass 的完整流程

用户定义模型（LlamaForCausalLM 等）
↓
vLLM 用 torch.compile 包装 forward
↓
TorchDynamo 追踪 → 原始 FX Graph
↓
PassManager 依次应用：
SequenceParallelismPass → AsyncTPPass → FusionPass → ...
↓
优化后的 FX Graph
↓

↓

3.7 一个比喻：装修房子

传统方式vLLM 图改写方式
每种户型（模型）单独设计装修方案所有户型先画一张标准毛坯图（FX Graph）
装修图纸里直接写"这里要用大理石"毛坯图就是毛坯图，不管用什么材料
换材料要重画所有户型由装修队（Pass）按规则扫图替换
不同师傅的活儿会打架各装修队按顺序进场，互不冲突

核心概念速查表

概念解释类比
FX GraphPyTorch 把模型表示成的计算图电路原理图
Node / Op图中的一个算子电路里的一个元件
Edge / Tensor算子之间流动的数据元件之间的导线
Subgraph大图中的一小块相连节点电路里的一个功能模块
Pattern要匹配的子图"模板"找元件的规格书
Pass对图的一次优化操作一个专门的装修工种
Subgraph Rewriter执行 Pattern 匹配和替换的工具自动化装修机器人
TorchDynamo追踪 Python 代码生成 FX Graph把手绘草图数字化
Inductor把优化后的图编译成 CUDA kernel把电路图变成实物
Symmetric Memory多 rank 共享的对称地址空间，支持 P2P 直读共享办公空间，谁都能拿东西
AllReduce所有 rank 持有完整 reduce 结果开会通报全员
ReduceScatterreduce 后每个 rank 拿不同分片分组讨论，各拿一部分结论
AllGather每个 rank 拿到所有 rank 的数据大家把作业交上来汇总
最终总结
  1. SP 在编译期把 AllReduce 改写为 ReduceScatter + AllGather，消除冗余 Norm/残差计算
  2. 这种改写的副产物是：通信原语变成了支持行级流水线的形式，为 AsyncTP 铺路
  3. AsyncTP 利用矩阵乘法的行独立性 + symmetric memory 的 P2P 能力，把通信和计算 overlap
  4. 整个机制通过 FX Graph + Pattern Matcher 实现，对模型代码完全透明，可灵活组合
